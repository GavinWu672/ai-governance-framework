# 2026-03-20

- Added a v2.6 governance-runtime design note and machine-readable draft under
  `docs/governance-runtime-v2.6.md` and
  `governance/governance_decision_model.v2.6.json`.
- The draft formalizes the enforcement model, decision ownership, policy
  precedence, evidence classification, violation handling, and determinism
  contract.
- The contract-repo boundary is now written down explicitly: preserve domain
  assets, but move verdict ownership and fallback behavior into the framework
  runtime.
- Recorded user feedback about governance friction in real collaboration:
  - validation tooling is fragile and expensive to maintain; many scripts are
    tightly coupled to implementation details, so small renames can break
    multiple validators
  - phase gates are too heavy for UI and exploratory work; agents often need to
    implement first and backfill governance artifacts later
  - multiple behavior contracts such as `AGENTS.md` and `governance/AGENT.md`
    create cognitive load and possible rule conflicts
  - schema changes trigger broad ripple effects across models, DTOs, APIs,
    frontend, and validators, causing repeated infrastructure-repair loops
  - long governance documents consume too much context budget and degrade room
    for code reasoning
- Recorded improvement directions from the user:
  - tiered governance, with L0 fast-track for UI and prototype work and stricter
    governance for core domain logic
  - decoupled validation focused on interface/schema outcomes rather than string
    scans or private implementation details
  - consolidated truth by reducing overlapping agent/governance documents and
    removing redundant backup artifacts
- Started turning that feedback into a formal status memo so the next governance
  simplification changes are driven by an explicit problem statement rather than
  only daily notes.

## Governance friction backlog

- Validation fragility:
  - more than 50 governance scripts now exist in `governance_tools/`
  - rename-level changes can break several validators at once
  - next action: audit validators that depend on file names, wording, or private
    implementation details
- Phase-gate weight:
  - `Analyze -> Define -> Test -> Implement` is too heavy for exploratory UI and
    layout work
  - next action: define a true L0 fast-track path for prototype and
    presentation-heavy work
- Document overlap:
  - `AGENTS.md` and `governance/AGENT.md` both act like live behavior contracts
  - next action: choose one canonical contract and reduce the other to a
    reference boundary
- Schema ripple effect:
  - additive model changes still force broad synchronized updates across DTOs,
    APIs, frontend, and validators
  - next action: define compatibility rules for additive schema changes and
    shift validators toward versioned normalized envelopes
- Context cost:
  - long governance docs consume too much context before implementation starts
  - next action: reduce default governance payload size and load deeper policy on
    demand

## Review notes on commit c71e0c1

- User review: commit `c71e0c1` clearly responds to the previously identified
  architecture gaps, but mostly at the definition/specification layer rather
  than full runtime implementation.
- What the review considers meaningfully addressed:
  - mixed enforcement is now described as a transitional seam rather than the
    final architecture
  - runtime is explicitly defined as the single decision computation source
  - three existing matrices were upgraded and a violation-handling matrix was
    added as machine-checkable spec
  - policy precedence is now explicit rather than implicit
  - evidence handling now includes validity/trust dimensions, not just presence
  - violation handling is modeled for abnormal/error states
  - a determinism contract now exists
  - integration boundaries for external contract repos are explicitly split into
    preserve/rewire/keep-out areas
- What the review considers still incomplete:
  - the change is still a decision-model draft, not a full runtime rewrite
  - decision ownership is clearer for computation and partial authority, but not
    yet fully modeled across computation/authority/presentation as separate
    sources
  - evidence flow is moving toward a push-first envelope model, but provenance
    fields and a full envelope contract are not yet fully specified
  - trace requirements are acknowledged, but a concrete machine/human trace
    schema is not yet present
  - verdict lifecycle/state-machine behavior is still not fully defined
- User assessment:
  - the commit is worth doing because it stabilizes the control plane for future
    runtime refactors
  - it helps control Kernel/USB contract impact by explicitly documenting the
    preserve vs rewire boundary
  - it does not yet complete execution-layer migration
- User-specified next validation targets:
  - runtime should start reading `governance/governance_decision_model.v2.6.json`
  - hooks/CI/validator paths should move final verdict ownership into runtime
  - trace artifacts and verdict artifacts should become formal outputs
  - legacy `hard_stop_rules` should be downgraded into policy input rather than
    continuing as a direct outcome switch
